# 065 — Apple AI sidecar: summarize and extract-tags endpoints

**Workstream:** W12
**Tag:** AFK
**Blocked by:** #064
**User stories:** US-1, US-36

## Behaviour

The Apple AI sidecar exposes `POST /v1/summarize` and `POST /v1/extract-tags`. Each receives text and returns a structured response using `LanguageModelSession` with `@Generable` response types. These are the high-volume, quality-tolerant tasks routed away from Claude: generating `body_summary` for events and producing tag candidates for atom classification.

## Acceptance criteria

- [ ] `POST /v1/summarize` accepts `{text: String, max_sentences: Int = 3}` and returns `{summary: String}` generated by a Foundation Models session.
- [ ] `POST /v1/extract-tags` accepts `{text: String}` and returns `{tags: [String]}` (3–5 relevant tags).
- [ ] Both endpoints use `@Generable` structs for type-safe LLM output parsing.
- [ ] `LanguageModelSession` is created per request (or cached per process if Foundation Models supports session reuse — follow the Apple docs).
- [ ] `swift test` passes unit tests for the response schemas (no real LLM call; mock the session).
- [ ] Integration test (marked skip in CI): call both endpoints with a fixture text and assert the response has the correct shape.

## Notes

`@Generable` and `LanguageModelSession` are the FoundationModels framework APIs on macOS 26. Follow Apple's documentation for the correct session lifecycle.

The unit tests should test the Vapor routing and response schema, not the LLM output (which is non-deterministic). Use a mock `LanguageModelSession` that returns fixture data.

Response time target: < 2 seconds per call for a short text (< 500 tokens). The on-device model is much faster than Claude for short tasks.
